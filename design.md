## 1. コンセプト・ゴール

### ターゲット
個人経営または小規模の美容サロン（ネイル・エステ・まつエク・ヘアサロン等）
主な利用者: **オーナー1名＋顧客**（スタッフ複数人の管理は将来拡張）

### 解決する課題
| 現状の課題 | Charm+ で解決 |
|-----------|-------------|
| 予約はLINEや電話で口頭管理 | 顧客がWebフォームから24時間予約可能 |
| 顧客情報がノート・頭の中 | デジタルカルテで施術履歴一元管理 |
| 売上・経費はExcelや手書き | 自動集計・青色申告連携 |
| ツールがバラバラで二重入力 | **予約→来店→売上→カルテまで自動連携** |
| お客様への連絡がバラバラ | 予約確認・リマインダーを自動送信 |

### 名称
**Charm+**（チャームプラス）
"あなたのサロンの魅力（Charm）に＋αを"

### 3-2. コード分割の方針（重要な改善点）

現在の `app.js` は2,655行・1ファイル。オールインワン化すると5,000行を超える見込み。
**モジュール分割が必須。**

```
charm-plus/
├── index.html              … オーナー管理画面（タブ構成）
├── booking.html            … 顧客向け予約フォーム
├── style.css               … 共通デザインシステム（CSS変数）
│
├── js/
│   ├── core.js             … 定数、GasAPI、Storage、Format（共通基盤）
│   ├── data.js             … Data / Templates / Allocation / AutoRules / Master
│   ├── customers.js        … 顧客管理モジュール
│   ├── booking-admin.js    … 予約管理（管理画面側）
│   ├── finance.js          … 売上・経費・帳簿・CSV
│   ├── charts.js           … Chart.js グラフ描画
│   ├── ui.js               … タブ制御・モーダル・トースト・共通UI
│   └── app.js              … init() エントリポイント（各モジュールを結合）
│
├── js/booking.js           … 予約フォーム専用ロジック
│
├── gas.gs                  … GAS バックエンド（全API）
├── design.md               … この設計書
└── setup_guide.md          … セットアップ手順書
```

> **注**: `<script>` タグの読み込み順で依存関係を解決する。
> ビルドツールは使わない（ターゲットユーザーが自分でホストするため）。

### 3-3. GASの制約と対策

| 制約 | 値 | 対策 |
|-----|---|------|
| 実行時間上限 | 6分/回 | 1回のAPI呼び出しは軽く保つ。バッチ処理は避ける |
| ペイロードサイズ | ~50MB | 画像はDrive直接保存（base64は1枚ずつ） |
| Sheetsの行数上限 | 100万行 | 小規模サロンでは問題なし（年間数千行程度） |
| 同時アクセス | GASはシングルスレッド | 予約の競合はタイムスタンプで楽観的排他制御 |
| MailApp送信上限 | 100通/日（無料） | 小規模サロンには十分 |

### 5-1. メニュー管理（設定タブ内）

**なぜ最初に作るか:**
- 予約フォームのメニュー選択に必要

- カルテの施術記録に必要

**UI:**
- 設定タブ内に「メニュー管理」セクションを追加
- メニュー一覧（並び替え対応）
- 追加/編集/無効化フォーム

---

### 5-2. 顧客管理（新規タブ: 「顧客」）

**機能:**
- 顧客一覧（五十音順 or 最終来店順）
- 検索（名前・フリガナ・電話番号）
- 顧客詳細画面:
  - 基本情報（名前、連絡先、アレルギー等）
  - 来店履歴（予約一覧から自動集約）
  - カルテ一覧
  - 売上サマリー（累計・客単価）
  - 次回来店提案日（最終来店 + メニューの推奨間隔）
- 新規顧客の手動登録
- **予約時に電話番号で自動マッチ** → 既存顧客ならリンク、新規なら自動作成

**判断ポイント:**
- 電話番号を一意キーとする（日本の美容サロンの慣習に合致）
- 予約フォームからの自動登録で、オーナーの手入力を最小化

---

### 5-3. 予約管理

#### 5-3-a. 顧客向け予約フォーム（`booking.html`）

**ホスティング方法の選択肢:**

| 方法 | メリット | デメリット |
|-----|--------|----------|
| GAS doGet() で HTML配信 | URL一つで完結、デプロイ不要 | GASの応答速度に依存 |
| 静的HTML（ローカル or GitHub Pages） | 高速表示 | 別途ホスティングが必要 |

> **推奨: GAS doGet() でHTMLを配信する方式。**
> オーナーにとって最も簡単。URLを1つ共有するだけで予約フォームが動く。
> GAS側で `HtmlService.createHtmlOutputFromFile()` を使用。

**予約フロー:**

```
1. 顧客がURLにアクセス
   ↓
2. メニュー一覧を表示（メニューマスタから取得）
   ↓
3. メニューを選択 → 日時選択カレンダーを表示
   - 既に埋まっている枠はグレーアウト
   - 営業時間外は選択不可
   ↓
4. 名前・電話・メール・備考を入力
   ↓
5. 確認画面 → 送信
   ↓
6. GAS側で予約レコード作成（status: pending）
   ↓
7. 顧客に確認メール送信（「予約を受け付けました。確定メールをお待ちください」）
   ↓
8. オーナーに通知メール送信（「新しい予約が入りました」）
```

**日時選択の仕組み:**
- 営業日・営業時間はマスタデータで管理（例: 月〜土 10:00-19:00）
- 予約の枠（スロット）は `duration` 分単位で区切る
- 既存予約と時間が重なるスロットは選択不可
- **1人サロンの前提: 同時間帯に1件だけ**

#### 5-3-b. 管理画面側（予約タブ）

**UI:**
- **日次ビュー**: タイムライン形式（当日のスケジュール）
- **週次ビュー**: カレンダー形式
- 予約カード: 時間、顧客名、メニュー、ステータスバッジ
- ステータス遷移ボタン:
  ```
  pending（未確認）→ confirmed（確定）→ completed（完了）
                                      → cancelled（キャンセル）
                                      → noshow（無断欠席）
  ```

**完了時の自動処理（★コア連携ロジック）:**

```
予約を「completed」に変更した瞬間:
  1. 取引シートに売上レコードを自動追加
     - type: income
     - amount: 予約の price
     - category: 売上
     - description: "#{顧客名} #{メニュー名}"
     - appointmentId, customerId をセット
  2. 顧客の lastVisit, visitCount, totalSpend を更新
  3. カルテの雛形を自動作成（オーナーが後から詳細を記入）
```

## 6. 開発フェーズ計画（改訂版）

### 依存関係に基づく順序

```
Phase 1: UIリデザイン＋バグ修正
   ↓ （見た目が整い、ベースが安定する）
Phase 2: メニューマスタ＋営業設定
   ↓ （予約・顧客管理の前提データ）

Phase 4: 予約フォーム＋予約管理

Phase 5: 統合（予約→売上→カルテの自動連携）＋ダッシュボード刷新
```

## 7. 技術的な判断と制約

| 項目 | 方針 | 理由 |
|-----|------|------|
| 認証 | 管理画面: URLの秘匿のみ（Phase 1）→ 簡易パスワード（Phase 5以降） | 個人利用のためURLが漏れなければ十分。将来的にはGASのWebアプリ設定で制限可能 |
| 予約フォーム認証 | 不要（公開URL） | 顧客がログインなしで予約できることが必須 |
| 予約の競合制御 | 楽観的ロック（作成時に重複チェック） | GASはシングルスレッド実行のためロック不要 |
| 予約フォームのホスト | GAS doGet() | オーナーの追加設定なしでURLが発行される |
| メール送信 | GAS MailApp | 追加のサービス契約不要。1日100通で十分 |
| LINE連携 | Phase 5以降で検討 | LINE Messaging API + GASで実現可能だが、LINE公式アカウント（月額0〜5,000円）が必要。まずはメールで最小限を実現 |
| PWA対応 | Phase 5以降で検討 | manifest.json + Service Worker でホーム画面追加可能に |
| ビルドツール | 使わない | ターゲットユーザーが `<script>` タグだけで使えることが重要 |

## 8. 現行バグ・改善リスト（Phase 1で対応）

- [ ] **フィルター不具合**: 収支一覧のフィルターが一部機能しない
